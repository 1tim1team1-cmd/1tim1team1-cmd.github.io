0) Проект: зачем и для кого
Назначение сайта (3 цели):

Личное портфолио‑хаб: хранить и структурировать творчество (музыка/3D/видео/игры), быстро добавлять и просматривать, убрать “хлам” с диска.
Витрина для заказчиков услуг (монтаж, музыка, 3D, разработка): показать качество → привести к заказу.
Для подписчиков/аудитории: отдельные вкладки по направлениям, удобное потребление контента.
CTA:

Главный: Заказать (в /services и/или глобально видимый контакт/кнопка).
Вторичный: Просмотреть (портфолио/музыка/видео) + “подписаться/связаться”.

“Liquid glass / Apple‑like minimalism”, современный clean UI.
“Живость” через микровзаимодействия и реакцию фона (orbs) на музыку.

5) Глобальная архитектура аудио
В Layout:

<audio id="main-audio" preload="metadata" transition:persist></audio> — не сбрасывается при навигации.
Глобальное состояние “что играет”:

window._tt_nowPlaying
window._tt_setNowPlaying(payload) → dispatch nowplayingchange
window._tt_getNowPlaying()
Ended routing:

В Layout один раз биндится audio.addEventListener('ended', ...)
Если существует window._tt_onEnded (задаёт активная страница), он вызывается; иначе — mpNext() (MiniPlayer).
MediaSession + hotkeys:

MediaSession metadata обновляется по nowplayingchange
Hotkeys: Space play/pause, стрелки seek ±5/15, media keys next/prev.

7) Theme system (Light/Dark) — финальная логика
Цель: единая тема для всего сайта, без мигания (FOUC), с сохранением в localStorage.

Где: Layout.astro

Ранний inline‑скрипт в <head> выставляет класс на <html> и colorScheme.
Дальше IIFE “theme system”:
toggleTheme() меняет localStorage и классы на html/body
initTheme() применяется на:
astro:page-load
astro:after-swap ← критически важно для View Transitions
Баг и фикс:

Белая шапка/кнопки после переходов (особенно /music и /video) исправилась добавлением:
document.addEventListener('astro:after-swap', initTheme);
CSS-токены темы:

В global.css переменные glass/text/page-bg переключаются в dark.
В идеале переменные держать на :root.theme-dark (html), чтобы не зависеть от body при swap.

9) UX-манифест (принятые принципы)
Витрина без трения: за 10–15 секунд понятно кто автор, что умеет, где примеры, как заказать.
Единый паттерн на вкладки: list/grid → detail card (часто sticky), активный элемент, быстрые действия (share/copy), deep‑link.
Apple-like минимализм: 1 главный объект + 1 основное действие + 1 вторичное.
Живость как feedback: toast/подсветки/микропереходы — подтверждают состояние, а не просто “украшают”.
A11y: удобная клавиатура, focus-visible, aria‑атрибуты, reduced motion.

13.2. TT Core Helpers (глобальный набор утилит для всех страниц)
Цель: убрать дубли кода с каждой вкладки: base-path, abs() для GH Pages, share/copy, scroll с prefers-reduced-motion, escape HTML.

Где: Layout.astro (важно: размещать до <slot />, чтобы страницы могли пользоваться этим объектом в своих inline-скриптах).

API:

TT.basePrefix() — определяет базовый префикс для GH Pages (/repo/)
TT.abs(path) — строит base-aware URL к /public/...
TT.isReducedMotion() — prefers-reduced-motion
TT.smartScrollIntoView(el, {block}) — авто “smooth/auto”
TT.toast(msg, {ms}) — глобальный toast
TT.copyToClipboard(text) — копирование (с fallback)
TT.shareOrCopy({url,title}) — navigator.share → иначе clipboard + toast
TT.escHtml(str) — экранирование текста для template strings
Критичный фикс, который уже учтён:

В inline JS нельзя использовать TypeScript-синтаксис (например (): string => {}) — вызывает Unexpected token ')'.
Поэтому весь TT Core — чистый JavaScript.
Ожидаемый вывод:

Страницы (/music, будущий /video, /3d) используют одну базу TT.*.
Меньше копипасты, меньше ошибок, быстрее перенос паттерна.

Примечание по GitHub Pages и регистру путей
На Windows пути не чувствительны к регистру, а на GH Pages — чувствительны.
Если ты создаёшь public/Videos, то URL будет /Videos/..., и TT.abs('Videos/...') должен совпадать регистром. Лучше заранее стандартизировать папки в нижнем регистре (public/videos) — но это решение можно принять перед кодом.
